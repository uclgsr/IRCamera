This document describes the previous `TimeSyncManager` implementation. The active solution pairs
`gsr/network/TimeSyncClient.kt`, `TimelineClock`, and the PC controller
`time_sync_service.py` (UDP + calibration API). Refer to the diagrams in
`docs/diagrams/time-sync-sequence.md` and `docs/diagrams/time-sync-components.md` for the
current design. The remaining sections are preserved for historical context.

================================================================================
                    TIME SYNCHRONIZATION FLOW (IMPLEMENTED)
================================================================================

COMPONENTS INVOLVED:
-------------------
1. RecordingService - Manages recording lifecycle and network connections
2. TimeSyncManager - Handles sync protocol and timestamp offset management
3. NetworkServer - Sends/receives messages to/from PC
4. ProtocolHandler - Processes incoming protocol messages
5. TimestampManager - Applies offset to all sensor timestamps
6. PC Controller - Master clock that coordinates all devices

KEY IMPLEMENTATION POINTS:
-------------------------

1. SYNC_INIT Message:
   - New protocol message (MSG_SYNC_INIT = "SYNC_INIT")
   - Sent by phone to request PC to initiate sync
   - No parameters needed

2. RecordingService Integration:
   - Sets up syncTriggerCallback in onCreate()
   - Callback sends SYNC_INIT via networkServer.sendMessage()
   - Called automatically at session start

3. TimeSyncManager:
   - performSessionStartSync() now triggers actual sync (not just logs marker)
   - Calls syncTriggerCallback.onManualSyncRequested()
   - Applies offset to TimeManager and TimestampManager

4. Automatic Triggers:
   - Session start: Always attempts sync
   - Periodic: Every 5 min if session > 10 min
   - Manual: Via TimeSyncManager.triggerManualSync() API

5. Offset Application:
   - timeManager.setClockOffsetFromProtocolSync(offsetNs, rtt)
   - TimestampManager.setClockOffset(offsetMs)
   - All subsequent timestamps use synchronized time

6. Logging:
   - Sync events logged to timesync_log.csv
   - Includes: t1, t2, t3, offset, rtt, quality, retry count
   - Quality based on RTT (EXCELLENT/GOOD/ACCEPTABLE/POOR)

================================================================================

WHAT PC CONTROLLER MUST IMPLEMENT:
----------------------------------

1. Handle SYNC_INIT:
   - When receiving "SYNC_INIT" message
   - Immediately send "SYNC_REQUEST t_pc=<current_time_ms>"

2. Handle SYNC_RESPONSE:
   - Parse "SYNC_RESPONSE t_pc=<T1> t_ph=<T2>"
   - Capture T3 = current_time_ms
   - Calculate: offset = T2 - ((T1+T3)/2), rtt = T3-T1
   - Send "SYNC_RESULT t1=<T1> t2=<T2> t3=<T3> offset=<offset> rtt=<rtt>"

See pc-controller/docs/TIME_SYNC_IMPLEMENTATION.md for detailed guide.
