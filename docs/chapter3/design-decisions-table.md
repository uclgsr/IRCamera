# Sensor Integration Design Decisions

## Table 3.2: Critical Design Decisions and Alternatives

This table captures critical design decisions with alternatives considered and rationale for each choice.

## Architectural Patterns Applied

| Pattern | Where Applied | Benefits | Trade-offs |
|---------|--------------|----------|-----------|
| **Repository Pattern** | Data access layer (Android) | Abstracts storage implementation, enables testing with mock repositories | Extra abstraction layer |
| **Observer Pattern** | Sensor state changes, UI updates | Decouples components, enables reactive UI | Can lead to callback complexity |
| **Command Pattern** | Network protocol (PC ↔ Android) | Encapsulates requests as objects, supports undo/redo, logging | More classes to manage |
| **Singleton Pattern** | TimeManager, SessionManager | Single source of truth for time and session state | Global state can complicate testing |
| **Factory Pattern** | Sensor creation (ThermalRecorder, GSRRecorder) | Encapsulates instantiation logic, supports different sensor types | Extra indirection |
| **State Machine Pattern** | Recording lifecycle (Android) | Clear state transitions, easier to reason about behavior | Requires careful state design |
| **Facade Pattern** | Sensor APIs (wraps complex SDKs) | Simplifies complex subsystems, consistent interface | Can hide useful SDK features |
| **Adapter Pattern** | Shimmer SDK integration | Adapts external SDK to internal interfaces | Extra translation code |

## Design Principles Applied

1. **Separation of Concerns**: Network, sensors, storage, UI in separate modules
2. **Don't Repeat Yourself (DRY)**: Shared utilities (TimeManager, Protocol), reusable components
3. **You Aren't Gonna Need It (YAGNI)**: Only implement required features, no speculative additions
4. **Keep It Simple (KISS)**: Simple protocols, standard formats, minimal abstractions
5. **Fail-Safe Defaults**: Sensors disabled if not available, continue with partial data
6. **Convention Over Configuration**: Sensible defaults (session directory, network port), minimal required configuration
7. **Modularity**: Independent sensor modules, pluggable architecture for new sensors
8. **Testability**: Interfaces for mocking, dependency injection where practical
